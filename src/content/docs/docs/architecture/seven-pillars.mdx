EDDI's architecture is governed by seven core principles. These aren't marketing bullet points — they are **hard architectural constraints** that shape every design decision in the codebase.

## 1. Configuration-Driven Logic, Not Code-Embedded Logic

agent rules, lifecycle pipelines, tool definitions, and routing rules are all defined as **JSON configurations**, not compiled Java code.

This means:
- Prompt engineers can iterate on agent rules **without touching Java**
- Changes are applied instantly via REST API or the Manager UI — **no recompilation**
- Configurations are version-controlled, diffable, and auditable

> **The mantra:** *Logic is Configuration, Java is the Engine.*

## 2. Deterministic Governance of Non-Deterministic AI

LLMs are inherently non-deterministic. EDDI wraps them in deterministic guardrails:

- **Budget caps** prevent runaway API costs
- **Circuit breakers** halt infinite loops and hallucination spirals
- **Audit trails** record every LLM call with tokens, cost, timing, and tool calls
- **Execution hashes** detect and terminate recursive agent traps

## 3. EDDI is the Engine, Not the Application

EDDI is **middleware** — it sits between your applications and AI providers. It is not a agent, not a SaaS product, and not a consumer app.

Your applications consume EDDI's REST APIs or MCP tools. EDDI orchestrates the AI. Your application owns the UX.

## 4. Security as an Architectural Foundation

Security is not a feature — it is a **structural property** of the codebase:

- **No `eval()`** — no dynamic code execution, no sandbox escapes
- **Vault integration** — secrets never appear in configurations
- **URL validation** — all external URLs are validated against allowlists
- **Path traversal protection** — all file operations are sandboxed
- **OIDC/Keycloak** — enterprise authentication out of the box

## 5. Enterprise-Grade Concurrency

AI orchestration is I/O-bound — agents spend most of their time waiting for LLM responses. EDDI leverages:

- **Java 25 Virtual Threads** — millions of lightweight threads, no OS thread exhaustion
- **Quarkus** — non-blocking reactive architecture that prevents carrier thread pinning
- **NATS JetStream** — distributed messaging for horizontal scaling

## 6. Transparent Observability

Every step in EDDI's pipeline is observable:

- **Audit ledger** — immutable, per-task log entries with LLM details, timing, cost, and tool calls
- **Pipeline logs** — server-side error and performance logging filtered by agent/conversation/level
- **CQRS telemetry** — execution state is separated from observability state

## 7. Progressive Disclosure in UX

EDDI's interfaces are designed for both business users and power users:

- **Business users** see clean, high-level agent management in the Manager UI
- **Power users** access raw JSON configs, pipeline definitions, and API endpoints
- **The API surface** starts simple (`setup_agent` creates everything in one call) and reveals complexity only when needed
