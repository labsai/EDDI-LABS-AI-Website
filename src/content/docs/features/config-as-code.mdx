## What's Configurable?

<CardGrid stagger>
  <Card title="Rules" icon="setting">
    Conditional response logic: "If the user mentions billing, set action to `redirect_billing`." Evaluated before the LLM call, controlling which output templates fire.
  </Card>
  <Card title="LLM Configuration" icon="puzzle">
    LLM provider, model ID, temperature, max tokens, tool bindings, RAG data sources. Switch from GPT-4 to Claude in seconds.
  </Card>
  <Card title="Output Sets" icon="document">
    Response templates with HTML formatting, quick reply buttons, action triggers. The full presentation layer is configuration.
  </Card>
  <Card title="HTTP Calls" icon="external">
    REST API integrations as pipeline tasks. Call your backend, webhooks, or third-party APIs — configured, not coded.
  </Card>
  <Card title="Workflows" icon="list-format">
    Pipeline composition — which tasks run, in what order. Add sentiment analysis by adding a pipeline extension; remove it by removing a line.
  </Card>
  <Card title="Agent Triggers" icon="rocket">
    Intent-to-agent routing rules for managed conversations. Route `customer_support` intent to Agent A in production, Agent B in staging.
  </Card>
</CardGrid>

## Immutable Versioning

Every resource in EDDI is **immutably versioned**. When you update an LLM config, EDDI creates version 2 while preserving version 1.

- **Instant rollback** — revert to any previous version via a single API call
- **A/B testing** — run two agent versions simultaneously with different configs
- **Audit compliance** — every change is tracked with its full history
- **Zero-risk updates** — if version 2 breaks, version 1 is untouched
