## Why AI Agents Love EDDI

<CardGrid stagger>
  <Card title="🤖 39 MCP Tools" icon="puzzle">
    EDDI exposes its **entire capability surface** via the Model Context Protocol. Claude Desktop, Cursor, Windsurf, and custom agents can create agents, manage conversations, deploy configurations, and read logs — all without touching the codebase directly.
  </Card>
  <Card title="📐 Predictable Patterns" icon="setting">
    Every resource follows the same CRUD pattern: `create → read → update → delete → deploy`. Once an AI agent learns one domain (agents), it can apply the same pattern to workflows, conversations, and triggers.
  </Card>
  <Card title="📝 Self-Documenting APIs" icon="open-book">
    MCP tool schemas include **detailed parameter descriptions**, required/optional flags, and example values. An AI agent can discover what tools exist and how to call them without reading source code.
  </Card>
  <Card title="🧩 Modular Architecture" icon="list-format">
    Clean workflow boundaries mean AI agents can focus on one domain at a time. Changing agent management code doesn't require understanding the conversation engine. Changes are **locally scoped** and testable.
  </Card>
</CardGrid>

## Agent-Friendly Codebase

<CardGrid>
  <Card title="Consistent Naming" icon="approve-check-circle">
    Classes, methods, and fields follow strict naming conventions. `AgentStore`, `AgentService`, `AgentResource` — the pattern is always `{Domain}{Layer}`. AI agents can predict file locations and class names from context alone.
  </Card>
  <Card title="Small, Focused Classes" icon="approve-check-circle">
    Average class size under 200 lines. Single responsibility principle enforced throughout. An AI agent can read and understand an entire class in one context window.
  </Card>
  <Card title="Typed Configuration" icon="approve-check-circle">
    Agent logic is **JSON configuration**, not compiled Java. AI agents can modify agent rules — prompts, pipelines, tool definitions — via REST API or MCP without understanding Java internals.
  </Card>
  <Card title="Comprehensive Error Messages" icon="approve-check-circle">
    API errors include structured error codes, descriptive messages, and correlation IDs. When an AI agent's action fails, the error response contains enough context to self-correct.
  </Card>
</CardGrid>

## The MCP Advantage

EDDI's MCP server is the **primary interface for AI coding agents**. Instead of parsing source code, agents can:

<CardGrid stagger>
  <Card title="Discover" icon="magnifier">
    `list_tools` → See all 39 available operations with full parameter schemas. No documentation crawling needed.
  </Card>
  <Card title="Build" icon="setting">
    `setup_agent` → Create a fully configured agent in a single MCP call: LLM provider, system prompt, tools, pipeline steps.
  </Card>
  <Card title="Test" icon="rocket">
    `chat` / `chat_managed` → Send messages and validate responses programmatically. AI agents can run their own acceptance tests.
  </Card>
  <Card title="Debug" icon="open-book">
    `read_logs` → Inspect pipeline execution, LLM token usage, tool call sequences, and timing data. Full observability without `grep`-ing log files.
  </Card>
</CardGrid>

## Frequently Asked Questions

<details>
<summary><strong>Can an AI coding agent modify EDDI's agent configurations?</strong></summary>

Yes — this is EDDI's primary design goal. Agent logic (prompts, pipelines, tool definitions) is **JSON configuration**, not compiled Java code. AI agents can read and modify configurations via the REST API or MCP tools (`update_agent`, `update_regulardictionary`, etc.) without any Java knowledge. Changes take effect immediately — no recompilation or redeployment needed.

</details>

<details>
<summary><strong>How does EDDI's MCP server help coding agents?</strong></summary>

EDDI's MCP server exposes 39 tools that cover the full agent lifecycle: creation, configuration, deployment, conversation, scheduling, and diagnostics. AI assistants like Claude Desktop or IDE plugins (Cursor, Windsurf) can interact with EDDI programmatically. The MCP tool schemas include detailed parameter descriptions, making them self-documenting — agents can discover capabilities without reading source code.

</details>

<details>
<summary><strong>Is the EDDI codebase easy for AI tools like Copilot or Cursor to understand?</strong></summary>

Yes. EDDI's codebase is designed with AI-readability in mind: consistent naming conventions (`{Domain}{Layer}`), small focused classes (under 200 lines average), typed configuration models, CDI dependency injection (no hidden singletons), and **1,000+ tests with zero failures** ensuring safe refactoring. The modular workflow structure means agents can work on one domain without needing the full codebase in context.

</details>

<details>
<summary><strong>What's the difference between using MCP tools vs. editing EDDI's source code?</strong></summary>

MCP tools let agents **operate** EDDI (create agents, manage conversations, read logs) without touching Java code. This is the recommended approach for 90% of use cases. Editing source code is only needed for adding new capabilities to the platform itself — such as implementing new pipeline task types or adding new MCP tools. The modular architecture ensures that even source-level changes are locally scoped and independently testable.

</details>
